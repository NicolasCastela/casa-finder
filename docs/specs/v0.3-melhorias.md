## Visão Geral

Quatro melhorias na ferramenta de busca de imóveis pra acelerar a triagem (user tem 5 dias pra alugar): (1) ordenação por bairro como secundária, (2) workflow centrado em **exclusão persistente** de bairros, (3) cache local com refresh manual, (4) preferência por tipo de imóvel como bônus no score, sem virar filtro.

## Motivação / Contexto

User está em busca real de aluguel em Campo Grande/MS, prazo apertado.
Pontos de atrito do estado atual:
- Cada run leva 2-3 min (não dá pra iterar rápido em filtros)
- `--neighborhoods` é whitelist; user prefere "tira esses ruins, deixa o resto"
- `--property casa-terrea` filtra; user quer "prefiro casa-térrea, mas mostra sobrado se for bom"
- Não dá pra agrupar visualmente por bairro pra planejar visitas no mesmo dia

## Research Findings

**Stack:** TypeScript + Node ESM + tsx + commander, sem framework. Estrutura flat.

- `format.ts` — `printTerminal`, `toJson`, `toCsv`, `toHtml` (HTML já tem toolbar JS de sort)
- `fetcher.ts` — curl wrapper

**Padrões a seguir:**
- ESM com extensões `.js` em imports (sourceful)
- Tipos centralizados em `types.ts`, sem default exports
- Pesos do score declarativos em `filter.ts`
- State persistido em `state.json` ao lado de `package.json`
- `slugify(...)` pra comparar bairros (já usado em `applyFilters`)

**Breaking Changes:**
- Nenhuma. Todos os campos novos em tipos são opcionais. State.json antigo carrega normal (campos faltantes → defaults).

## Riscos e Mitigações

| Nível | Risco | Mitigação |
|---|---|---|
| Médio | Cache serve dados velhos e user perde imóveis novos sem perceber | Loga `📦 cache: X hits / Y fresh` em cada run; `--refresh` invalida tudo |
| Baixo | Cache cresce indefinido em disco | Adicionar `--cache-clear`; notar tamanho típico no README (~3MB pra 60 imóveis enriquecidos) |
| Baixo | Slugify não bate "Centro" salvo vs "centro" do API | `state.ts` + `applyFilters` aplicam `slugify` em ambos os lados (já é o padrão) |

## Requisitos Não-Funcionais

- **Performance:** Cache hit total deve responder em <5s (vs 2-3 min sem cache)
- **Observabilidade:** Logar `cache hits/fresh` por fase, bairros bloqueados aplicados, e bônus de preferência aplicado

## User Stories

- Como caçador de aluguel, quero **ordenar por bairro** pra agrupar candidatos vizinhos e visitar vários no mesmo dia
- Como caçador de aluguel, quero **manter blacklist de bairros que persiste entre runs**, sem precisar lembrar a lista toda vez
- Como caçador de aluguel, quero **rodar várias vezes ao dia sem refetch**, refinando filtros sobre os mesmos dados
- Como caçador de aluguel, quero **dar preferência a casa-térrea** sem perder de vista sobrados (caso apareça um sobrado excelente)

## Considerações de Arquitetura

- **Decisão:** Cache infinito (sem TTL automático), refresh exclusivamente manual via `--refresh`.
  **Motivo:** User pediu explicitamente "só quando eu precisar". TTL automático adiciona regra invisível e reintroduz o problema atual em pequena escala.
  **Alternativa rejeitada:** TTL de 24h em listagens — adiciona comportamento implícito, user perde controle.

- **Decisão:** Cache key = `<list|detail>-{sha1(url).slice(0,16)}.json`.
  **Motivo:** URL captura todos os params. Hash evita caracteres inválidos no FS Windows.
  **Alternativa rejeitada:** Filename humano-legível baseado em params — frágil em Windows e Linux com nomes longos/acentuados.

- **Decisão:** Preferência por tipo é uma category nova no score (max 8 pts), separada das outras 8 categorias.
  **Motivo:** Mantém retrocompatibilidade — runs sem `--prefer` mantêm scores idênticos. Breakdown fica explícito.
  **Alternativa rejeitada:** Sub-categorizar dentro de "layout" — confunde o breakdown e dilui a sinalização.

- **Decisão:** `--block "X"` aceita um bairro por chamada e adiciona à lista. Idempotente (não duplica).
  **Motivo:** Comando simples e previsível. User pode encadear `--block "A" && --block "B"`.
  **Alternativa rejeitada:** `--block "A,B,C"` — ambíguo com bairros que têm vírgula no nome.

- **Decisão:** O bloqueio é só uma blacklist. Sem auto-sugestão baseada em listings que o user marcou como ❌ no HTML.
  **Motivo:** HTML já tem ❌ por imóvel (localStorage do browser). Misturar com blocklist persistida do CLI traria duas fontes de verdade.

## Follow-up resolvido — Bug de paginação fake da API

**Descoberto:** durante uso real, user notou que busca retornava ~10-20% dos imóveis esperados.

**Causa raiz:** A API `/imoveis` declara `pagination: {page, total, pages}` mas **ignora silenciosamente os params `page` e `limit`**. Sempre retorna no máximo 20 listings por request, qualquer que seja a página pedida. O campo `meta.seed` muda entre requests mas o conjunto retornado é determinístico pra mesmos filtros.

**Solução:** bisecção recursiva por faixa de preço. Quando `pagination.total > data.length`, divide o range `[priceMin, priceMax]` ao meio até cada slice retornar tudo (ou hi-lo ≤ R$50). Cada slice tem cache key próprio (URL distinta).

**Resultado:** mesmo filtro (`casa-terrea`, R$2000-4500, 2+ quartos), antes 11 imóveis, depois **220 imóveis**. Custo: ~50 requests adicionais na primeira run (cache subsequente).

**Implementação:** `infoimoveis.ts` — funções `fetchSlice()` e `fetchAllForType()`. `fetchPage()` agora chama `fetchAllForType()` 1× por tipo e retorna `hasNextPage: false` (a API não pagina de fato). Flag `--max-pages` virou no-op mas mantida pra retrocompat.
